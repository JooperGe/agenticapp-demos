# 02 · 比赛机制（裁判状态机）

> 目标：让"比赛机制健全"落地。裁判是**规则的权威**，负责治理比赛流程与合法性，但**从不指挥球员的具体动作**。它的全部输出是：更新自身状态机 + 向 EventBus 发布事件。

## 1. 裁判的职责边界

裁判**做**：
- 维护比赛阶段状态机（下节）。
- 观察世界，判定离散事件：出界、进球、（后续）越位/犯规。
- 判罚归属：球权判给哪一队、判定定位球类型（界外球/角球/球门球/任意球）。
- 依规则**选定发球球员**，广播 `RestartAssigned`。
- 判定发球者是否就位（`RestartReady`）、比赛是否恢复（`PhaseChanged→InPlay`）。

裁判**不做**：
- 不告诉任何球员"你去哪、你踢给谁"。
- 不移动球或球员。
- 不修改球员决策。

> 选定发球球员 ≠ 指挥。裁判只是宣布"这个界外球归 #5 发"（就像真实裁判指认发球方），**#5 如何走过去、发给谁，是 #5 自己的决策**。

## 2. 比赛阶段状态机（Match FSM）

```
                    ┌────────────┐
                    │  PreMatch  │  22 人入场就位
                    └─────┬──────┘
                          │ 全员到位
                          ▼
                    ┌────────────┐   开球被踢动且离开中圈
                    │  Kickoff   │ ──────────────────────┐
                    └─────┬──────┘                        │
                          │                               ▼
                          │                        ┌────────────┐
                          └───────────────────────▶│   InPlay   │
                                                   └─────┬──────┘
                     ┌───────────────────────────────────┤ 球出界/进球/犯规
                     ▼                                     │
              ┌──────────────┐                            │
              │   DeadBall   │  子类型：                   │
              │              │  ThrowIn / Corner /         │
              │              │  GoalKick / FreeKick /      │
              │              │  Penalty / KickoffAfterGoal │
              └──────┬───────┘                            │
                     │ 发球者就位并合法开球                  │
                     └───────────────────▶ InPlay ◀────────┘

    HalfTime（45:00）→ 换边、恢复 → Kickoff(对方)
    FullTime（90:00）→ 结束
```

状态存于 `World.phase`，形如 `{ main: 'DeadBall', sub: 'ThrowIn', restartTeam, restartSpot, takerId }`。
每次 `main`/`sub` 变化，裁判发布 `PhaseChanged`，让全体球员据此重新决策站位。

## 3. 死球检测与定位球归属（rules/outOfPlay.js）

判定输入：球的连续位置轨迹 + `Ball.lastTouch{playerId, teamId}` + 球场几何。

### 3.1 出界判定（几何）

- **越过边线（touchline，球场长边）** → 界外球（ThrowIn）。
- **越过底线（goal line，球场短边）**：
  - 若球完全越过**门框之间**且在横梁下 → **进球**（见 3.3）。
  - 否则 → 角球或球门球（见 3.2）。

判定"完全越过"：用球心是否越过线 + 球半径容差（`rules.json.lineTolerance`）；对高速球做**线段-线相交**插值，取真实越线点作为 `exitPoint`（避免因帧步长漏检/穿透）。

### 3.2 定位球类型与球权（谁的球）

| 情形 | 类型 | 球权判给 | 发球位置 |
|------|------|----------|----------|
| 越过边线 | 界外球 ThrowIn | 最后触球方的**对方** | 越线点（贴边线） |
| 越过底线，最后触球=**进攻方** | 球门球 GoalKick | 防守方 | 防守方小禁区内 |
| 越过底线，最后触球=**防守方** | 角球 Corner | 进攻方 | 最近的角球点 |
| 球越过门线且在门框内 | 进球 Goal | — | 中圈重新开球 |

"进攻方/防守方"由**越过的是哪一侧底线**决定：越过 A 队防守的底线，则 A 是防守方。

### 3.3 进球判定（rules/goal.js）

球**整体**越过门线、且在两门柱之间、横梁之下 → `GoalScored{teamId, scorerId}`。
`scorerId = Ball.lastTouch`（后续可细化乌龙球）。随后进入 `DeadBall:KickoffAfterGoal`，球权判给**失球方**。

## 4. "由哪个球员发球"的选定规则（rules/restartTaker.js）

裁判在确定球权后，用一套**可配置规则**选定发球球员，并广播 `RestartAssigned{playerId, restartType, spot}`：

- **界外球 / 角球 / 任意球**：选**发球方中距离 `spot` 最近的非门将球员**（可配置：某些角球指定专职主罚者）。
- **球门球**：默认选**门将**（可配置为后卫）。
- **中圈开球**：选**发球方指定的开球手**（默认中场核心，如 #6/#10）。

选定逻辑是纯函数：`(world, restartType, spot, roster) → playerId`。默认策略"最近者优先"，后续可被战术配置覆盖（例如"角球固定由 #7 主罚"）。

> 就位判定：裁判每帧检查 `takerId` 是否进入 `spot` 容差圈内且速度足够低 → 发布 `RestartReady{takerId}`，允许该球员产出发球 `KickIntent`。在此之前，`IntentResolver` 会**驳回**任何踢球意图（死球期间球不可被踢动），保证机制健全。

## 5. 合法性预检（IntentResolver 与裁判协作）

物理层的 `IntentResolver` 在执行意图前做**合法性预检**，规则边界由裁判状态定义：

- 死球阶段：只有 `takerId` 且已 `RestartReady` 时，其 `KickIntent` 才被放行；其余球员的 `KickIntent` 一律驳回（可移动）。
- 开球：对方球员进入中圈/9.15m 内 → 该越线为非法，可回退或（后续）判重开。
- 冲突仲裁：多名球员同帧对同一松球产出争抢意图 → 由属性 + 距离 + 时机计算胜者，其余判定为失败/碰撞。

## 6. 首个里程碑（垂直切片）里裁判的最小实现

为验证核心范式，垂直切片**只需**：

- FSM：`PreMatch → Kickoff → InPlay → DeadBall:ThrowIn → InPlay`。
- 规则：`outOfPlay` 只做**边线出界 → 界外球**；`restartTaker` 只做"最近非门将者"；`goal` 可先占位。
- 暂缓：角球、球门球、越位、犯规、点球（结构预留，后续填充）。

这样即可跑通你的界外球例子闭环，同时架构对后续规则**开放扩展**（新增一个 `rules/*.js` + 一个事件类型即可）。
