# 00 · 现有设计的根因诊断

> 目的：说清楚现有设计"差强人意"的**根本原因**，而不是罗列表面 bug。新设计正是针对这些根因重构的。

## 一句话结论

**现有系统是"以帧循环为中心、以配置分层为骨架"的系统；而你想要的是"以球员智能体为中心、以事件感知驱动、由健全的裁判状态机治理"的系统。主客颠倒了。**

现有的 8 层架构（物理/规则/属性/技能/决策/状态/策略/分析）是按**知识领域**切的**静态配置分层**，回答的是"系统有哪些知识"；但它没有回答运行时的关键问题：**"谁，在什么时刻，感知到了什么，如何决策，如何驱动下一步。"** 这正是你反复强调的"球员是一切动作的核心和起源"所要求的。

## 根因逐条拆解（附代码证据）

### 根因 1：球员不是"动作的起源"，而是被中央循环"拉取"的函数

`js/player-agent.js:362` 的 `simulateFrame()` 是一个中央编排器，每帧对所有球员调用 `makeDecision()`：

```
allActions = allPlayers.map(player => player.makeDecision(gameState, ...))
```

这意味着球员只有在**被轮询时才"思考"**，本质仍是"系统拉动球员"，而非"球员感知到事件后自发行动"。`config/8-execution/engine.json` 里那套"6 阶段循环"把这一点固化成了架构核心——**循环是主角，球员是配角**。

你要的恰恰相反：开球指令下发 → **开球手自己感知到** → 他**自己决策**下一步。动作的因果起点必须在球员身上。

### 根因 2：设计里有"事件驱动"，代码里完全没有

`docs/03-realtime-decision.md` 花了大篇幅描述事件总线、P0–P4 优先级、事件中断决策——这套思路是对的。但**代码里没有任何事件总线**。`player-agent.js` 和 `engine.json` 全是定时轮询，事件驱动只停留在文档。**设计与实现严重脱节**，导致"球员感知事件"这一核心能力根本不存在。

### 根因 3：比赛机制（裁判）是空壳，死球闭环缺失

- `player-agent.js:405` 的 `validateActions()` 直接 `return true`——**裁判不判任何东西**。
- `getVisibleEnvironment()`（`:391`）返回空数组——**感知层未实现**，球员实际上"什么都看不到"。
- 出界检测、"由哪个球员发球"的判定、发球球员的就位、其他球员的重新站位——**全部缺失**。

你举的"界外球"例子（系统规则判定发球球员 → 该球员感知到 → 去球的位置 → 其他人重新站位）在现有代码里没有任何一环存在。**"比赛机制健全"无从谈起。**

### 根因 4：规则/技能/策略是全局共享的，不属于球员个体

`plans/mellow-giggling-quiche.md` 已诊断出 `demo/index.html` 的问题：AI 行为由全局 `DEFAULT_RULES`（角色 × 情境 → 字符串条件 → 字符串动作）定义，用 `new Function` 执行。后果：

- 规则是全队共享的，**不是每个球员自己的**——与"球员是核心"直接矛盾。
- 字符串条件 + `eval` 难以可视化、难以调试、易出错。
- "做什么（策略）"和"做得怎么样（技能）"混在一起。

### 根因 5：关注点耦合，无法演进

`demo/index.html` 单文件 690 行，物理 / AI / 渲染 / UI / 比赛流程全部耦合。改一处牵动全身，谈不上"分层可编辑可进化"。

### 根因 6：分层是"描述性"的，不是"可执行"的

`ARCHITECTURE.md` 的 8 层是很好的**领域知识清单**，但它不是运行时架构：

- 它没定义**模块之间在一帧内如何交互、数据如何流动**。
- "第 4 层决策"和"第 8 层执行"的边界模糊：决策到底是球员发起还是引擎发起？文档两种说法都有（`ARCHITECTURE.md:27-37` 说分布式智能体，但 `engine.json` 又是中央循环拉取）。**架构自相矛盾。**

## 新设计如何逐条回应

| 根因 | 新设计的对策 |
|------|-------------|
| 1 球员被拉取 | 球员是唯一的 **Intent 产出方**；MatchLoop 只调度，不决策 |
| 2 无事件总线 | 引入 **EventBus**；裁判广播事件，球员订阅并中断式响应 |
| 3 裁判空壳 | **Referee FSM** + 出界/定位球判罚规则，产出完整死球闭环 |
| 4 全局规则 | 每个球员**独立持有**感知、黑板、效用决策器 |
| 5 耦合 | ES Modules 按职责拆分：core / world / referee / agent / physics / render |
| 6 分层矛盾 | 用**运行时职责**（观察→感知→决策→意图→仲裁→物理→事件）替代静态领域分层 |

> 注：旧的领域知识（属性、技能、物理参数、战术）并没有被丢弃，而是**降级为数据/配置**，喂给新的运行时结构；它们不再是架构骨架。
