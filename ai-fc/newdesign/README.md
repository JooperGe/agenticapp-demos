# AI FC — 重构设计（newdesign）

本目录是对 `ai-fc` 现有设计与代码的**推倒重来**。它不复用旧的 8 层配置骨架，而是围绕一个核心主张重新组织整个系统：

> **球员（Player Agent）是一切动作的唯一起源；比赛由一个健全的裁判状态机（Referee FSM）治理；两者通过事件总线（Event Bus）解耦。帧循环只是调度器，不做任何决策。**

## 核心范式（一句话）

- **裁判只观察、判定、广播事件**——它决定"什么合法、球权归谁、由谁发球"，但**从不指挥任何球员做具体动作**。
- **球员只感知事件与局部形势、自主决策、输出意图（Intent）**——传球、跑位、发球、就位，全部由球员自己发起。
- **物理层只执行意图、仲裁冲突**——它不含任何"战术智慧"。

用户提出的三个例子（开球、面对阻击、界外球判罚与就位）在本设计中都是同一条链路的自然结果，而非特判脚本。详见 `01-architecture.md` 的"三个时序流"。

## 阅读顺序

1. `00-diagnosis.md` — 现有设计为什么"差强人意"（根因，非表面症状）。
2. `01-architecture.md` — 新运行时架构、模块划分、事件模型、数据流、三个端到端时序流。
3. `02-match-mechanics.md` — 裁判状态机与"健全比赛机制"（死球/定位球/开球/出界/判罚归属）。
4. `03-player-agent.md` — 球员智能体内部：感知 → 效用决策（Utility AI）→ 意图。
5. `04-data-schemas.md` — Event / Intent / World / LocalModel 的字段契约。
6. `PLAN.md` — 落地计划（首个里程碑：垂直切片）。

## 本轮已对齐的边界决策

- **范围**：只聚焦"比赛仿真引擎"（场上 22 人 + 裁判 + 物理 + 渲染 + 调试面板）。训练/养成/赛季/商业化**暂不涉及**。
- **决策模型**：球员用 **Utility AI（效用系统）**——每个候选动作实时算分取最优，天然贴合"实时结合形势判断"。
- **技术形态**：**模块化原生 JS（ES Modules）+ Canvas**，多文件按职责拆分。
- **首个里程碑**：**最小垂直切片**——开球 → 推进 → 出界 → 界外球判罚 → 指定球员就位并发球 → 恢复比赛。用最短闭环证明核心范式成立。规模 **11v11**，感知先用全场可见占位，**1 倍速**真实节奏。

## 待细化的开放问题（后续对话逐条敲定）

M1 相关的边界已敲定（11v11 / 全场可见占位 / 1 倍速 / 数据契约已定，见 `04-data-schemas.md`）。以下留待后续里程碑：

1. 感知的"不完美性"细度：视野扇角 + 距离衰减 + 估计误差的具体参数（M3 引入）。
2. Utility 的 consideration 曲线是否做成可视化可编辑（M5 策略编辑器）。
3. 确定性/可复现：种子化随机数（建议 M1 即引入，`core/Rng.js` 已在架构中）。
4. 越位/犯规何时接入（M4）。
