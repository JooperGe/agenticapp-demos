# AI FC — 性能分析与部署架构

> 头脑风暴第四轮：计算量分析、事件风暴处理、离线 vs 在线、技术方案

## 一、核心认知：球员AI不是大模型

球员在场上的"AI"本质是一套**确定性规则引擎 + 数值系统**，不需要LLM或神经网络推理。

```
球员"AI"的本质：

  输入（数值）          处理（规则）           输出（数值）
  ┌──────────┐      ┌──────────────┐      ┌──────────┐
  │ 球的位置   │      │  if-else树    │      │ 移动方向   │
  │ 队友位置   │ ──→ │  权重计算     │ ──→ │ 移动速度   │
  │ 体力值     │      │  随机偏差     │      │ 是否射门   │
  │ 情绪值     │      │  优先级排序   │      │ 传球目标   │
  └──────────┘      └──────────────┘      └──────────┘

  整个过程没有"思考"，只有计算
  每个球员每帧的计算量 ≈ 几百次加减乘除 + 几次条件判断
```

## 二、规则引擎 vs LLM 对比

| 维度 | 规则引擎AI（本方案） | LLM API |
|------|---------------------|---------|
| 单次决策延迟 | 0.01-0.1ms | 500-3000ms |
| 单次计算量 | 几百次算术运算 | 数十亿次参数运算 |
| 确定性 | 可控的随机偏差 | 不可控的生成结果 |
| 离线运行 | 原生支持 | 必须联网 |
| 22人同时 | 2ms内全部完成 | 需要22次API调用，10+秒 |
| 电池消耗 | 几乎为零 | 巨大（网络+发热） |

## 三、事件风暴处理机制

比赛瞬息万变，但系统有三层过滤机制：

### 第一层：空间过滤

```
球场上的事件不是每个人都接收
- 球在左路 → 右路后卫不关心
- 球在进攻三区 → 己方门将低频关注
- 对方犯规 → 距离20m外的队友不触发响应
每个球员只处理"与自己相关的"事件
```

### 第二层：优先级过滤

```
同一时刻多个事件到达 → 只处理最高优先级
同时收到：
  [队友传球给我]  P1
  [对手靠近逼抢]  P1
  [教练说压上去]  P2
  [队友在前方跑位] P3
→ 只处理前两个P1，P2/P3排队或丢弃
→ P1的两个事件合并为"被逼抢中接球"
```

### 第三层：频率控制

```
持球人：    每帧决策（必须反应快）
附近球员：  每3帧决策
远端球员：  每10帧（甚至每20帧）
团队协调器：每2秒一次
一帧内最多只有1-2个球员在"思考"
```

## 四、真实计算量估算

```
一帧（60fps）的总计算量：

  物理引擎：
    球的运动更新：          O(1)       → ~10次算术运算
    22个球员的位置更新：     O(22)      → ~220次算术运算
    碰撞检测（网格加速）：   O(k)       → ~50次比较
    小计：                              ~300次运算

  感知系统（分频）：
    本帧需要更新感知的球员： ~6-8人
    每人扫描周围实体：       ~10-15个
    小计：                              ~100次运算

  AI决策（分频）：
    本帧需要决策的球员：     ~2-3人
    每人决策树遍历：         ~50-100个节点
    每个节点：权重计算+比较   → ~10次运算
    小计：                              ~300-1000次运算

  事件处理：
    本帧产生的事件数：       ~2-5个
    每个事件路由+过滤：       ~20次运算
    小计：                              ~100次运算

  总计：每帧约 800-1500次简单算术运算

  对比参考：
    iPhone 15 单核性能：~1万亿次浮点运算/秒
    1500次运算耗时：< 0.001ms
    一帧预算（60fps）：16ms

  → AI计算只占帧预算的不到 0.01%
  → 瓶颈在渲染，不在AI计算
```

## 六、为什么核心AI必须是离线的

### 延迟问题
- LLM API延迟：500ms-3000ms
- 比赛需要：每帧16ms完成22人决策
- 差了300倍，根本不可行

### 成本问题
- 一场比赛：22人 × 平均每5帧决策 = 142,560次API调用
- 即使每次100 token，每场也要$2+
- 用户一天10场 = $20/天，完全不可持续

### 可控性
- LLM可能输出不合逻辑的行为
- 规则引擎行为在预设范围内，比赛结果可复现

### 离线体验
- 用户在地铁上、飞机上也能玩
- 核心体验不依赖网络

## 七、LLM的合理使用场景（非实时、非核心）

### 赛后报告（异步）
```
prompt示例：
"你是足球教练。我的球队2-1赢了，但控球率只有42%。
 前锋F射门5次只进了1个，体力在第60分钟降到40%。
 中场A传球成功率91%但跑动距离全场最少。
 对方的进球来自我方右路防守空档。
 请给出训练建议。"
```

### 球探报告（异步）
生成新球员时，本地生成属性数据 → LLM生成描述性报告

### AI教练对话（用户主动触发）
用户在训练界面点击"问AI教练" → 弹出对话框，完全异步

## 八、最终技术方案

```
核心引擎（C++ / Rust / Unity）：
  • 物理引擎：自定义轻量级
  • AI决策引擎：决策树 + 效用系统
  • 属性/情绪/体力系统：数值计算
  → 全部本地运行，不依赖任何API

游戏框架（Unity / Unreal / Godot）：
  • 渲染、UI、音效、存档系统

后端服务（可选）：
  • 云存档、PvP异步对战、数据更新推送

AI增强（可选）：
  • 赛后报告、球探报告、教练对话助手
  → 用LLM API，用户可关闭
```
